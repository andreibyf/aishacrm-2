## 1. Purpose and scope

Phase 3-14 closes the Phase 3 + Slice 2 arc by recording the one decision Phase 3 was designed to produce: **may Phase 4 production pilot planning begin?**

The decision being made here is **not** "may we activate a production pilot." Phase 3 + Slice 2 deliberately stopped at the staging boundary: every runbook is dry-run-by-default, no live execution occurred, no production endpoint or credential exists anywhere in the repository. A separate Phase 4 design + review cycle is required before _any_ production-side work begins — and Phase 4 activation itself requires its own go/no-go decision packet downstream of Phase 4-0 design freeze. This packet decides the **earlier**, **less risky** question: whether the team is authorized to begin designing Phase 4.

Inputs to this decision:

- **The deferred / unresolved items** honestly enumerated in Phase 3-13 §8: persistent-events route lift (gating prerequisite for Phase 3-10 execution), 10 operator-side execution evidence packs still empty (3-2 through 3-11), Phase 4 production pilot decision (this packet), 6 known limitations.

Phase 3-14 does **not** scope Phase 4 itself. The §9 sequence is a recommended ordering for the Phase 4-0 design freeze cycle, not a Phase 4 plan.

## 3. The decision

### 3.1 Pilot planning vs pilot activation — definitions used by this packet

| Term                                    | What it includes                                                                                                                                                                                                                                                                                                                                                        | What it does **NOT** include                                                                                                                                                                                                                                                                                                                                                                                                       |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Phase 4 production pilot planning**   | The Phase 4-0 design freeze cycle: scope decisions per §8 (pilot tenant, provider path, migration / apply window, rollback owner, monitoring owner, support owner, evidence pack owner); design freeze document(s); Codex design review; the §9 recommended planning sequence runbook authoring. **All doc-only work, no env change, no migration, no provider write.** | Any production Doppler (`prd_prd`) mutation; any production tenant selection-and-flip; any production Coolify app provisioning; any production migration; any production ERPNext / other provider OAuth or credential provisioning; any production observability wiring; any production traffic routed at the Finance v2 route. **None of this is Phase 4 _planning_; all of it is Phase 4 _activation_** and is gated separately. |
| **Phase 4 production pilot activation** | The operator-side execution of whatever Phase 4 planning produces — actual production Doppler changes, actual production migration apply, actual production ERPNext sandbox / production tenant connection, actual production traffic. The first time any production-side asset is touched.                                                                             | Phase 4 planning. The two are sequential; pilot planning produces the runbooks that pilot activation executes (mirroring the Phase 3 dry-run-then-execute pattern).                                                                                                                                                                                                                                                                |

This packet is about **pilot planning** only. **Pilot activation is gated by its own, separate go/no-go cycle (Phase 4-N where N is the activation-decision packet) downstream of Phase 4-0 design freeze + Codex review + the §7 preconditions being satisfied.**

### 3.2 Recommendation: **GO-WITH-CONDITIONS** for Phase 4 production pilot **planning**.

**Justification:** Phase 3 + Slice 2 have produced everything Phase 4 planning needs as input:

- A complete, Codex-cleared staging activation runbook arc (Phase 3-1 through 3-11).
- A complete, Codex-cleared adapter runtime (Slice 2A through 2E).
- A consolidated local-verification baseline (390/390 automated + Andrei's manual run + 18 cite-confirmed security boundaries).
- Honest documentation of deferred items, both as direct blockers on Phase 3-10 execution (the persistent-events route lift) and as Phase 4 scope inputs (per-tenant credential router, audit worker, credential encryption, etc.).

Phase 4 _planning_ itself is doc-only work — design freeze authoring, Codex design review, runbook authoring. It carries the same risk profile as Phase 2C / Slice 2-0 / every Phase 3-x runbook authoring task: doc work that doesn't touch any environment. The team has demonstrated 12 + 6 successful packets of exactly that pattern. Authorizing Phase 4 planning carries minimal risk.

**The conditions** (§6) attach because pilot _activation_ downstream depends on operator-side execution that has not yet happened (Phase 3-13 §8.2: 10 operator-side evidence packs still empty) and on a code slice that has not yet been designed (Phase 3-13 §8.1: persistent-events route lift). Phase 4 planning can proceed in parallel with that work; Phase 4 activation cannot precede it.

### 3.3 Recommendation is **NOT** GO (unconditional)

A pure GO would imply "Phase 4 planning may begin without conditions, and the conditions surfaced by Phase 3 are not blockers for activation either." Neither is true. The §6 conditions matter because they shape what Phase 4 planning _must_ produce — namely, runbooks that explicitly gate activation on §7 being satisfied. A pure GO would risk Phase 4 planning producing runbooks that quietly assume the staging arc has been executed.

### 3.4 Recommendation is **NOT** NO-GO

A NO-GO would imply "Phase 4 planning is premature" or "Phase 3 work is insufficient as planning input." Neither is true. Phase 3 + Slice 2 produced exactly the documentation surface a Phase 4-0 design freeze needs to consume. Halting Phase 4 planning until the §7 preconditions are operator-executed would serialize work that can safely run in parallel — the Phase 3 staging runbooks are operator-actionable today, and Phase 4 planning is design work that produces operator-actionable Phase 4 runbooks for downstream activation.

## 6. Conditions attached to the GO recommendation

Phase 4 planning is authorized **only if** the resulting Phase 4 design freeze + runbooks honor the following conditions. Codex's Phase 4-0 design review will be the verification point for each.

1. **Phase 4 planning is doc-only.** No runtime code, no env-var change, no migration, no provider write, no production action by any Phase 4-N planning task. Same hard-constraint pattern as Phase 3-x runbook authoring.
2. **Phase 4 production pilot activation is gated separately** by its own go/no-go decision packet (Phase 4-N where N is the activation-decision packet, mirroring this Phase 3-14 packet's role for Phase 3). Phase 4 planning does **not** activate anything; it produces the runbooks an activation packet will eventually execute against.
3. **`ENABLE_FINANCE_PERSISTENT_EVENTS` fail-closed route guard preserved.** Phase 4 planning does NOT propose flipping this flag without the prerequisite route lift slice landing, being Codex-cleared, and being staging-executed first. The Phase 4-0 design must explicitly chain the lift slice as a prerequisite, not bundle it.
4. **`FINANCE_PROVIDER_WRITES_ENABLED=false` default preserved.** Phase 4 planning does NOT propose a long-lived `true` state in any environment, staging or production. Any production-side provider write window MUST mirror Phase 3-10 §6.6's controlled-flip pattern: deliberate, time-bounded, mandatory revert, negative-proof verification.
5. **Sandbox-first.** Phase 4 planning starts with a production-side sandbox / staging-equivalent verification — not a direct flip to a production provider. The Phase 3-10 §6.6 controlled-flip pattern is the model.
6. **Single controlled tenant** for any Phase 4 pilot — same posture as Phase 3 (`a11dfb63-...` is the staging controlled tenant; the production pilot will pick exactly one production tenant, never N>1 in the initial pilot). Per-tenant explicit allow-list, no implicit "all" fall-through.
7. **No production credential or production endpoint** appears in any Phase 4 planning document. Same standard as every Phase 3 doc: production-side identifiers are operator-supplied at execution time and never committed to git.
8. **No QuickBooks / Xero / NetSuite OAuth** unless Phase 4 scope explicitly adopts one of those adapters AND a corresponding adapter implementation slice has landed and been Codex-cleared. ERPNext-only is the default Phase 4 scope, mirroring the Slice 2A choice.
9. **No payment movement** by any Phase 4 task. Same hard-constraint pattern as 2C-9 §3.
10. **Phase 4 activation precondition table** (§7 below) is consumed by every Phase 4-N planning packet; each runbook must explicitly list which preconditions it requires before its activation step can run.

If a Phase 4 planning task ever proposes work that violates any of these conditions, Codex's design review surfaces it and the planning halts pending re-scope.
