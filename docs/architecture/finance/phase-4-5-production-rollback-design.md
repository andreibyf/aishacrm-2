## 1. Purpose and scope

Phase 4-5 freezes the production rollback contract for the Phase 4 pilot. Phase 4-0 §7 slice #6 + §9 + §11.2 row 6 require:

- A rollback owner **distinct** from the deploy owner (independently reachable).
- Rollback controls for every lever the activation flips (route gate, persistent-events flag, workers, tenant module gate, controlled tenant list, provider-write kill switch, ERPNext credentials).
- Abort criteria + evidence capture.
- No rollback procedure depends on destructive data deletion.

The pilot activation (Phase 4-21) is the first time any production-side asset is touched. If anything goes wrong — sustained 5xx, projection corruption, unauthorised provider write, RLS / tenant mismatch, customer-side complaint, security incident — rollback must be:

- **Fast** — measurable in minutes, not hours.
- **Authorised by a person who isn't the deploy owner** — a deploy owner outage or compromised account must not block rollback.
- **Reversible only forward, never destructive** — no data is dropped; the rollback returns the system to a known good prior state by reversing levers, not by deleting events.
- **Observable** — every rollback step emits signals per Phase 4-4 §3 signal 16 + audit-trail entries.

Phase 4-5 is design freeze only — no runbook execution language ("do step X then step Y on host Z"). The operational runbook downstream of this freeze owns the execution language; this packet owns the **contract** the runbook honors.

**Inputs:**

- Phase 4-0 §7 slice #6 + §9 owner matrix + §11.2 row 6 + §11.1 trigger #9 (no guardrail weakening) + §6.6 controlled-flip pattern (Phase 3-10).
- Phase 4-1 / 4-2 / 4-3 / 4-4 design freezes — define the levers Phase 4-5 must be able to reverse.
- The 16 safety guardrails ([`phase-3-activation-evidence-pack.md`](./phase-3-activation-evidence-pack.md) §7) — Phase 4-5 explicitly does not weaken any of them; many are themselves the rollback levers.

**Outputs of this packet:**

- §3: Rollback owner separation contract.
- §4: Rollback lever inventory + per-lever rollback step.
- §5: Abort criteria — what triggers a rollback decision.
- §6: Evidence capture — what gets recorded during and after a rollback.
- §7: Communication protocol during a rollback (consumes Phase 4-0 §9 escalation tree + Phase 4-4 §7 customer-impact map).
- §8: Non-destructive guarantee — the explicit list of operations forbidden in any rollback step.
- §9: Rehearsal contract — Phase 4-20 cannot verify slice #6 PASS without a rehearsal record per Phase 4-0 §11.2 row 6.
- §10: Hard constraints.
- §11: Acceptance.

**Phase 4-5 does NOT:**

- Author the operational rollback runbook. That runbook (covering ssh commands, Doppler edits, Coolify clicks, SQL queries) is a downstream Phase 4 packet.
- Implement any code (no rollback emission code, no audit-trail wiring, no observability hook). Phase 4-4 emission code lands when its implementation packet is dispatched.

## 3. Rollback owner separation

Phase 4-0 §6 condition #6 + §7 precondition #6 + §9 owner matrix require: **rollback owner distinct from deploy owner, reachable independently.**

| Aspect                          | Frozen by Phase 4-5                                                                                                                                                                                                                                                                                                                                              |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Distinct individuals.**       | Rollback owner and deploy owner MUST be two different people, with two different access paths (separate Doppler logins, separate Coolify accounts, separate SSH keys). Each has full authority to execute every step in §4 without the other's approval during an active rollback.                                                                               |
| **Independent reachability.**   | Phase 4-20 row 7 verifies this by independent contact-path check (the production owner matrix packet — Phase 4-0 §10.2's 4-7 slot under the post-`3a036d10` P2 #1 renumbering — captures: separate phone, separate email or Slack DM, neither routed through the other's device). If one is on vacation or unreachable, the other can still rollback.            |
| **Pre-authorised levers.**      | The rollback owner does NOT need the deploy owner's approval to execute any step in §4. The Phase 4-20 GO authorises the deploy owner to flip levers forward; the **same** Phase 4-20 GO pre-authorises the rollback owner to flip the same levers back. Once Phase 4-20 has cleared, both directions are authorised; rollback owner uses the reverse direction. |
| **Trigger autonomy.**           | Rollback owner may trigger rollback unilaterally if any §5 abort criterion fires. Rollback does NOT require monitoring owner or security reviewer concurrence to start; concurrence is captured during evidence capture (§6) but not blocking on the decision.                                                                                                   |
| **Communication obligation.**   | Rollback owner notifies monitoring owner + deploy owner within 5 minutes of rollback start (the §7 protocol). Notification is not a precondition; it is concurrent.                                                                                                                                                                                              |
| **Security reviewer override.** | A security reviewer (Phase 4-0 §9) may instruct rollback owner to execute a security-tier rollback (signals 13 / 14 from Phase 4-4) without other concurrence. Security takes precedence.                                                                                                                                                                        |
| **Stand-down handoff.**         | After rollback completes, rollback owner remains on-deck until a written stand-down is declared (within the Phase 4 evidence pack). Stand-down is declared by either: monitoring owner confirming signals are green for ≥ 30 min, or security reviewer confirming the incident is contained.                                                                     |

**Phase 4-20 verification** (row 7 of §11.2): rollback owner identified, reachable via separate channel, has all pre-authorised access at the Doppler / Coolify / SSH layer required for §4 steps.

**Ordering during a rollback** (the operational runbook downstream specifies exact sequence; Phase 4-5 freezes the **priority order**):

1. **First — stop the bleeding.** Whichever lever is producing the failing signal: route-flag (#4), provider-writes (#5), or workers (#7/#8) gets flipped first.
2. **Second — contain.** Flip the other above-the-fold levers to prevent further blast radius (e.g., if #5 caused the incident, also flip #4 and #8 to prevent in-flight queued writes from firing).
3. **Third — pause data flow.** Tenant module gate (#2) or controlled-tenant allow-list (#3) — stops new events from entering the pipeline.
4. **Fourth — revoke credentials.** Credentials (#6) — defence-in-depth; even if a worker is somehow still running, no credential lookup succeeds.
5. **Fifth — process-level shutdown.** `ENABLE_FINANCE_OPS` (#1) — last-resort full unmount.
6. **Migration revert (#9) is reserved for migration-level corruption only.** Day-to-day rollback does not revert migrations.

## 5. Abort criteria

A rollback decision is triggered when any of the following fire. Phase 4-5 freezes the criteria; the operational runbook downstream may refine numeric thresholds.

| #   | Criterion                                                                                                                                                                                                                                             | Trigger source                                                            | Authority to decide rollback                                      |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| 1   | **Sustained 5xx rate on Finance v2 route** (Phase 4-4 signal 1 — > 5% of requests for > 5 min).                                                                                                                                                       | Phase 4-4 observability / manual log inspection.                          | Rollback owner unilaterally; monitoring owner concurs.            |
| 2   | **Projection degraded for the pilot tenant** (Phase 4-4 signal 8 — any `projection_state.status = 'degraded'` row).                                                                                                                                   | Phase 4-4 signal 8.                                                       | Rollback owner; concur with deploy owner before final stand-down. |
| 3   | **Replay validation fails for any pilot-tenant projection** (Phase 4-4 signal 7).                                                                                                                                                                     | Phase 4-4 signal 7.                                                       | Rollback owner; concur with deploy owner.                         |
| 4   | **Sustained adapter sync failure for pilot tenant** (Phase 4-4 signal 9 — > 5 fails in 1 hour OR > 20 fails in 10 min).                                                                                                                               | Phase 4-4 signal 9.                                                       | Rollback owner; concur with deploy owner.                         |
| 5   | **Unauthorised provider write detected** (any `audit_events` of type `finance.adapter.sync_succeeded` OR `finance.adapter.sync_failed` recorded outside an authorised controlled-flip window).                                                        | Phase 4-4 signal 11 + audit_events SQL.                                   | **Security reviewer authority** — rollback fires immediately.     |
| 6   | **RLS / tenant mismatch error fires** (Phase 4-4 signal 13 — any single occurrence).                                                                                                                                                                  | Phase 4-4 signal 13.                                                      | **Security reviewer authority** — rollback fires immediately.     |
| 7   | **Credential lookup failure for pilot tenant** (Phase 4-4 signal 14 — sustained > 5 min).                                                                                                                                                             | Phase 4-4 signal 14.                                                      | Rollback owner; concur with security reviewer.                    |
| 8   | **Customer-side complaint matching a pilot-impact category** (per Phase 4-4 §7 + Phase 4-15 escalation triage).                                                                                                                                       | Support owner relays from customer-side primary contact.                  | Rollback owner; concur with support owner + deploy owner.         |
| 9   | **Deploy owner unavailable mid-activation** (deploy owner cannot complete activation; rollback returns the system to pre-activation state pending resumption).                                                                                        | Deploy owner self-declares OR monitoring owner observes lack of progress. | Rollback owner unilaterally.                                      |
| 10  | **Any safety guardrail violation observed** (16 guardrails from Phase 3-13 §7 — e.g., persistent-events flag flipped without the route lift implementation, provider-writes flag flipped outside a controlled window, sandbox-only URL guard bypass). | Phase 4-4 + manual review.                                                | **Security reviewer authority** — rollback fires immediately.     |
| 11  | **Migration apply produces unexpected schema state** (Phase 4-8 production migration runbook step fails).                                                                                                                                             | Deploy owner.                                                             | Deploy owner OR rollback owner; PITR restore option triggered.    |

**Phase 4-5 design constraint:** abort criteria are **expressly defined** — not "whenever something feels wrong." The runbook codifies the §5 list; deviations from the list require an explicit security-reviewer note in the evidence pack.

## 7. Communication protocol during rollback

Communication is concurrent with rollback execution, not blocking. Consumes Phase 4-0 §9 escalation tree + Phase 4-4 §7 customer-impact mapping.

| Phase                                                                    | Comm action                                                                                                                                                                                                              |
| ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Rollback decision — t0**                                               | Rollback owner posts in the AiSHA on-call channel: "Rollback initiated. Criterion: §5 row N. Stand by for lever flips." Tags monitoring owner + deploy owner + (if criterion 5/6/10) security reviewer.                  |
| **Lever flips — t0 to t0+10 min (target)**                               | Rollback owner executes §4 levers in §4 priority order. Each flip posts a one-line confirmation to the on-call channel.                                                                                                  |
| **Customer-side notification — t0+15 min (target, criterion-dependent)** | Support owner notifies customer-side primary contact per Phase 4-4 §7 mapping. For security incidents (criteria 5, 6, 10), customer-side notification may be delayed pending security reviewer's containment assessment. |
| **Stand-down decision — t0+30 min (earliest)**                           | Monitoring owner confirms Phase 4-4 signals are green for ≥ 30 min OR security reviewer confirms containment. Stand-down declared in on-call channel.                                                                    |
| **Post-rollback comms — t0+1 hour**                                      | Deploy owner posts a summary in the on-call channel: criterion, levers flipped, time-to-rollback, customer-side status, next steps (re-attempt cadence).                                                                 |
| **Root cause analysis — within 7 days**                                  | Deploy owner + security reviewer co-author a brief RCA, appended to the rollback evidence record. RCA gates re-activation: a fresh Phase 4-20 GO requires the RCA as input.                                              |

## 8. Non-destructive guarantee

**No rollback step in §4 deletes a row, drops a non-rolled-back table, truncates an event stream, or zeroes out an audit log.** The operational runbook MUST honor this.

| Forbidden operation in any rollback step                                                                                                             | Why                                                                                                                                                                                          |
| ---------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `DELETE FROM audit_events ...`                                                                                                                       | `audit_events` is append-only and immutable (migration 173 triggers enforce). Deletion violates immutability and destroys audit trail.                                                       |
| `DELETE FROM projection_state ...`                                                                                                                   | Projection state is recomputable from `audit_events`. Deleting projection state forces a replay on reactivation but loses the cursor history — not a rollback action, a different operation. |
| `DELETE FROM tenant_integrations WHERE id = $rollbackedRowId`                                                                                        | Phase 4-3 §8 + §10. Deactivating (`is_active = false`) is sufficient; deletion would destroy the credential row that may be needed for re-activation forensics.                              |
| `TRUNCATE` on any finance schema table.                                                                                                              | Same reason as DELETE — destroys data, violates the no-destructive rule.                                                                                                                     |
| `DROP TABLE` on any table introduced by migrations 172-175 or the Phase 4-3 encryption migration.                                                    | Migration revert is column-drop only per Phase 4-3 §8. Table drop is reserved for PITR restore (a separate path that operates on a snapshot, not the live database).                         |
| Provider-side cleanup that destroys customer-side records (e.g., deleting invoices from the customer's ERPNext that AiSHA created during the pilot). | Provider-side cleanup is the provider's contract (ERPNext draft-invoice void, journal reversal). Phase 4-5 does not authorise AiSHA to call provider-side DELETE endpoints in rollback.      |
| Forging or backdating any `audit_events` row to "hide" the activation having occurred.                                                               | Immutability triggers prevent the technical operation; the rule restates the principle for human readers.                                                                                    |

**The rollback's correctness depends on the no-destructive guarantee.** Reactivation must be able to read the full audit trail of the prior activation + rollback to inform whether to retry.
