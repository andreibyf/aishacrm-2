## 1. Purpose and scope

Phase 4-4 freezes the production observability + alerting design for the Finance Ops pilot. It adapts Phase 3-11's staging signal catalog ([`staging-observability-verification.md`](./staging-observability-verification.md)) for production-tier concerns: production-grade alert routing, on-call surfaces, customer-impact signals, and the IMPLEMENTED / PARTIAL / DEFERRED honesty marking that Phase 3-11 established.

Phase 3-11 captured 9 signals (worker up/down, worker lag, replay failure, degraded projection, adapter sync failure, dead-letter count, audit evidence build failure, RLS / tenant mismatch error, finance route 401/403 spikes) with explicit gaps. Phase 4-4:

- Carries forward Phase 3-11's catalog as the production baseline.
- Adds production-specific signals: provider-write kill-switch state, persistent-events route state, credential lookup failures, rollback events, security incident signals.
- Defines **alert severity** per signal — production alerts route to AiSHA's on-call surface (per Phase 4-0 §9 owner matrix), not just a logger.
- Defines **owner routing** per signal — which role (deploy owner, rollback owner, monitoring owner, support owner, security reviewer) is paged for each.
- Marks each signal honestly: IMPLEMENTED (today, in committed code), PARTIAL (data exists but no consumer / dashboard / alert), or DEFERRED (data does not exist yet).
- Defines dashboards / log views required before activation can be authorised.

**Inputs:**

- Phase 3-11 signal catalog ([`staging-observability-verification.md`](./staging-observability-verification.md) §4 — 9 signals).
- Phase 4-0 §9 owner matrix.
- Phase 4-1 §6 no-silent-fallback design (degraded signal).
- Phase 4-3 §9 credential audit log.
- The 16 safety guardrails ([`phase-3-activation-evidence-pack.md`](./phase-3-activation-evidence-pack.md) §7) — observability must make the guardrails' state observable.

**Outputs of this packet:**

- §3: Signal catalog — production-tier expanded, with implemented/partial/deferred + alert severity + owner routing per signal.
- §4: Alert severity and owner routing rules.
- §5: Dashboards / log views required before activation.
- §6: Honest enumeration of what is NOT yet implemented (gaps to close as separate slices).
- §7: Customer communication posture for the customer-facing impact dimension.
- §8: Required test surface for the alert-routing wiring (when implemented).
- §9: Hard constraints.
- §10: Acceptance.

**Phase 4-4 does NOT:**

- Wire any alert anywhere — no PagerDuty configuration, no Opsgenie configuration, no Slack channel webhook, no email alert.
- Provision any monitoring infrastructure — no log aggregator (Loki / Datadog / etc.), no metrics backend (Prometheus / Grafana / etc.).
- Touch production. Production observability provisioning is part of a future Phase 4 operator-side activation packet; Phase 4-4 freezes the design that packet implements.
- Add code that emits new signals. Signals are either already emitted by committed code (carried forward from Phase 3-11) or designed to be emitted by Phase 4-1 / 4-2 / 4-3 implementation packets. Phase 4-4 catalogs the signals; it does not introduce new emit sites.

## 4. Alert severity and owner routing rules

**Severity definitions** (production-tier; stricter than staging):

| Severity   | Response window                        | Routing                                                                                                         |
| ---------- | -------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `CRITICAL` | Page immediately (within 5 min).       | Phase 4-0 §9 escalation tree from monitoring → deploy → rollback. Security signals also page security reviewer. |
| `WARNING`  | Notify within business hours (1 hour). | Monitoring owner via the AiSHA on-call channel (Slack channel + email). No paging unless severity escalates.    |
| `INFO`     | Dashboard-only. Reviewed daily.        | No active routing. Monitoring owner reviews dashboard during pilot.                                             |

**Cross-role rules:**

- **Customer-impact signals** (signals 1, 3, 9, 13 for the pilot tenant) additionally notify the Support owner because the customer-side primary contact may surface symptoms before our internal signals do.
- **Security signals** (signals 11, 13, 14) additionally page the Security reviewer per Phase 4-0 §9.
- **Rollback signals** (signal 16) additionally notify the Rollback owner who initiated the rollback (acknowledgment loop).
- **Deploy owner** is paged for any CRITICAL until the Rollback owner explicitly takes over or stand-down is declared.

**Alert deduplication:** within a 10-minute window, the same signal does not page twice. The implementation packet for the alert wiring decides whether deduplication runs at the alert backend (Opsgenie / PagerDuty) or at our log aggregator.

## 7. Customer communication posture

The pilot tenant is a real customer. Observability must include the customer-facing impact dimension — signals that operators see internally translate (sometimes) into customer-side symptoms.

| Internal signal                           | Customer-facing symptom                                                                                                                   | Communication trigger                                                                                                                                                                                 |
| ----------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Signal 1 (route 5xx) sustained            | Finance Ops console shows errors; backend operations from the customer side fail.                                                         | Support owner notifies customer-side primary contact within 15 min of CRITICAL paging. Phase 4-15 designs the template.                                                                               |
| Signal 9 (adapter sync failure) sustained | Customer's ERPNext sandbox does not receive expected syncs.                                                                               | Support owner notifies customer-side technical contact within 30 min of sustained WARNING.                                                                                                            |
| Signal 13 (RLS / tenant mismatch)         | Potentially: another tenant could see this tenant's data, or this tenant could see another's. CRITICAL security signal.                   | Security reviewer leads the response; Support owner coordinates customer communication AFTER security reviewer assesses impact. Customer communication may be delayed until containment is confirmed. |
| Signal 14 (credential lookup failure)     | Sync paused for the affected tenant.                                                                                                      | Support owner notifies customer-side technical contact within 30 min of WARNING; CRITICAL within 15 min.                                                                                              |
| Signal 16 (rollback event)                | Finance Ops console may show "degraded" or "production activation not authorised" banner (UI Slice 1 §10 guardrail banners surface this). | Support owner notifies customer-side primary contact in advance if rollback is planned; immediately after if rollback was emergency.                                                                  |

**Phase 4-15 (production support + escalation design) consumes this matrix** and authors the actual customer-communication templates + the support owner's runbook. Phase 4-4 defines the trigger mapping; Phase 4-15 implements the response.
