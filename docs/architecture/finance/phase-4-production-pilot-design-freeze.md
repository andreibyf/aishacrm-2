## 1. Purpose and scope

Phase 4-0 freezes the **planning scope** for the Phase 4 production pilot under the Phase 3-14 GO-WITH-CONDITIONS recommendation. It does **not** activate anything. It does **not** authorise any production-side touch. It produces the contract every downstream Phase 4-N planning packet must honor and tells Codex exactly what to enforce against each subsequent Phase 4-N review.

The decision being made here is **not** "may we activate the production pilot." Phase 3-14 §3.2 already authorised Phase 4 production pilot _planning_, and Phase 3-14 §6 attached the conditions under which planning must proceed. Phase 4-0 operationalises those conditions into:

- Concrete pilot tenant selection criteria (§5).
- Concrete provider path commitments (§6).
- Concrete required code/design slices before activation (§7).
- Concrete required operator-side evidence before activation (§8).
- Concrete owner matrix with role separation (§9).
- Concrete Phase 4 planning sequence + parallel-safe packet grouping (§10).
- Concrete hard stop conditions + the explicit Phase 4-20 activation gate (§11).

Phase 4-0 does **not** make pilot tenant _selection_, owner _assignment_, or migration-window _scheduling_ — those are downstream Phase 4-N deliverables. Phase 4-0 freezes the **criteria, roles, sequence, and gates** so the downstream packets can be authored without re-deciding architecture mid-flight.

## 5. Candidate pilot tenant selection criteria

Phase 4-0 freezes the **criteria** for candidate selection. The actual candidate recommendation and customer authorisation capture are Phase 4-4 deliverables; Phase 4-0 does **not** name a tenant.

**A candidate pilot tenant must satisfy every criterion below**, evaluated against the production CRM state by the Phase 4-4 selection packet:

| #   | Criterion                                        | Why                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      | Operationalisation in Phase 4-4                                                                                                                                                                                                                                        |
| --- | ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Single controlled tenant**                     | Same pattern as Phase 3 staging — exactly one production tenant for the initial pilot; no N>1 cohort. Bounds the customer-facing blast radius and matches the controlled-tenant scope already enforced in code (`parseControlledTenantIds()` allow-list).                                                                                                                                                                                                                                                | Phase 4-4 produces a single recommended tenant UUID; no fallback list, no batch.                                                                                                                                                                                       |
| 2   | **Customer relationship maturity**               | The tenant must be an established customer of AiSHA with a stable point-of-contact already known to the AiSHA team. A fresh / trial tenant is not eligible — incident handling during a pilot needs an established communication channel, not a first-contact escalation.                                                                                                                                                                                                                                | Phase 4-4 documents the AiSHA-side relationship owner + the customer-side primary contact + customer-side technical contact.                                                                                                                                           |
| 3   | **Finance ops feature interest**                 | The tenant must have expressed business interest in the finance-ops feature surface (draft invoices, ledger, journal, approvals, adapter sync). Pilots imposed on uninterested tenants are coercion and produce uninformative signal.                                                                                                                                                                                                                                                                    | Phase 4-4 cites prior customer communication (Slack/email/call notes) demonstrating expressed interest.                                                                                                                                                                |
| 4   | **Pilot participation willingness**              | The tenant must explicitly opt in to the pilot. Opt-in is captured as a written authorisation (email or signed document) covering: scope of the pilot, sandbox-first progression, data handling, expected outage tolerance, rollback policy. Verbal-only authorisation is insufficient.                                                                                                                                                                                                                  | Phase 4-4 captures the written customer authorisation in a Phase 4 evidence pack table row; the production tenant UUID is recorded only after the authorisation is signed.                                                                                             |
| 5   | **Fallback-tolerance for short outages**         | The tenant must be able to tolerate at least one short controlled-flip window (matching Phase 3-10 §6.6's structure) where finance-ops UI may be temporarily degraded or read-only. Tenants whose business operations cannot tolerate any finance-ops downtime are not eligible for the initial pilot — they may be eligible for the post-pilot rollout (Phase 5).                                                                                                                                       | Phase 4-4 documents the tenant's operating-hours window and identifies a candidate flip window outside business hours.                                                                                                                                                 |
| 6   | **Business risk if pilot fails**                 | The tenant's financial impact from a failed pilot must be bounded and recoverable. The pilot must NOT route any actual customer-facing payment movement, statutory filing, or audit-trail-of-record event through the finance-ops surface during the initial window. Even if the production provider write window opens later, the first writes must be reversible (draft invoices that can be voided; journal entries that can be reversed via the existing reverse endpoint; no irreversible filings). | Phase 4-4 documents the bounded-impact rationale per scope category (invoices, journals, ledger, adapter sync, audit pack) and explicitly excludes payment-movement / statutory-filing categories from pilot scope.                                                    |
| 7   | **Sandbox availability (provider-side)**         | The tenant must be able to provide a sandbox ERPNext instance (or accept a Phase-4-provisioned sandbox) for the pre-flight progression. The pilot does NOT begin by routing to the customer's production ERPNext; it begins on a sandbox and only progresses to production ERPNext after sandbox-stage evidence is populated.                                                                                                                                                                            | Phase 4-4 documents the sandbox URL + credentials provisioning path (customer-provided sandbox or AiSHA-provisioned sandbox the customer connects to).                                                                                                                 |
| 8   | **Customer-side data-handling sign-off**         | The tenant must explicitly authorise the data flows the pilot introduces: which customer-side data (account names, deal amounts, contact names, invoice line items) crosses the AiSHA → ERPNext sandbox boundary; how long it is retained on each side; how it is deleted if the pilot is aborted. Same written-authorisation requirement as criterion #4.                                                                                                                                               | Phase 4-4 includes a data-handling addendum that the customer signs alongside the pilot authorisation.                                                                                                                                                                 |
| 9   | **No regulated industry hard-blockers**          | If the tenant operates in a regulated finance industry (broker-dealer, depositary, payments institution, regulated insurer) that would impose statutory audit obligations on AiSHA from day one of the pilot, the tenant is not eligible for the initial pilot — those regulatory paths require a separate Phase 5 review cycle. The default pilot tenant operates in an unregulated or lightly-regulated industry.                                                                                      | Phase 4-4 documents the tenant's industry and explicitly checks regulatory-blocker status.                                                                                                                                                                             |
| 10  | **Tenant is on the production CRM, not staging** | The pilot is by definition production; staging-only tenants are not eligible. The pilot tenant's tenant_id appears in the production Supabase project's `tenants` table.                                                                                                                                                                                                                                                                                                                                 | Phase 4-4 verifies the candidate tenant exists in the production `tenants` table without performing any production mutation (read-only check via the operator-supplied production credentials at Phase 4-4 execution time; Phase 4-0 does not perform any such check). |

**Tenants that fail any criterion are not eligible for the initial pilot.** They may be eligible for Phase 5 post-pilot rollout under that phase's own selection criteria.

**Bias rule:** Phase 4-4 selection rationale must explicitly avoid bias toward tenants with the _least_ business activity (which would minimise signal) or the _most_ business activity (which would maximise blast radius). The selection target is a "well-instrumented and motivated mid-volume tenant" whose pilot success is informative without being existentially risky.

## 7. Required code/design slices before activation

The Phase 3-14 §7 preconditions table identified concrete code work that must land before Phase 4 production pilot activation. Phase 4-0 formalises that list into a **code/design slice contract**: each row is a separately-designed-and-reviewed slice, NOT a sub-task of an activation runbook. Each slice is doc-only at the design-freeze stage; each becomes implementation-with-tests at its implementation packet; each is Codex-cleared before any production-side runbook may depend on it.

| #   | Slice                                    | Why required before activation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 | Phase 4 packet that delivers it | Activation can fire when                                                                                                                                                                                                                                               |
| --- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Persistent-events route lift**         | Today the Finance v2 route writes events to an in-memory store; the adapter worker reads Postgres. Without the lift the two never meet, so Phase 3-10 cannot execute end-to-end and Phase 4 cannot claim a real route-to-projection flow. Largest gap between Slice 2 deliverables and an executable Phase 3-10 (Phase 3-13 §8.1). Lift must NOT bundle the `ENABLE_FINANCE_PERSISTENT_EVENTS` flag flip in the same slice — flip is operator action gated separately.                                                                                                                                                                                                                                                         | Phase 4-1                       | Phase 4-1 design freeze + implementation + tests committed + Codex-cleared + staging-executed against Phase 3-10 §6.5.                                                                                                                                                 |
| 2   | **Projection-backed reads**              | Today the read-only Finance v2 endpoints (`/runtime/status`, `/journal-entries`, `/ledger`, `/profit-loss`, `/balance-sheet`) serve from the in-memory projection store regardless of `ENABLE_FINANCE_PERSISTENT_EVENTS`. With the route lift (slice #1) the read path must consume the Postgres-backed projection store consistently so reads and writes see the same world. Without this, the projection store backend a tenant gets depends on which worker last wrote — non-deterministic and untestable. Same packet may also expose the gap inventory the UI Slice 1 §8.2 documents (per-projection cursor endpoint, registered-adapter list, audit-events list) so the UI can render real signals instead of gap cards. | Phase 4-1 (paired)              | Phase 4-1 includes the read-path projection routing decision; readiness verified by Phase 4-11 production projection worker activation runbook + Phase 4-4 design freeze (observability) + the downstream operational observability runbook.                           |
| 3   | **Per-tenant ERPNext credential router** | Today the adapter worker reads ERPNext credentials from process env (one set per worker process). Production with multiple tenants on multiple ERPNext sandboxes/instances cannot work with one set of process-env credentials — the worker would either route every tenant to the same credentials or fail. Per-tenant routing via `tenant_integrations.api_credentials` (decision E4) is the multi-tenant production prerequisite. Must NOT introduce a code path that reads cleartext production credentials from JSONB at runtime — slice #4 (encryption migration) must land first or land together with mandatory at-rest encryption.                                                                                    | Phase 4-2                       | Phase 4-2 design + implementation + Codex-cleared + slice #4 either landed or co-shipped + staging-executed against a multi-tenant test fixture.                                                                                                                       |
| 4   | **Credential encryption migration**      | Decision E4 chose plain-JSONB storage as the POC posture, accepting that production credentials in plain JSONB are not acceptable. Phase 4 production cannot land production ERPNext credentials in plain JSONB. The encryption migration must apply before any production credential row is inserted. Includes: schema migration for at-rest encryption; key management decision (Doppler-supplied key vs KMS-supplied key); decrypt-at-read code path in the credential router (slice #3); rollback for the encryption migration; PITR snapshot policy before apply.                                                                                                                                                         | Phase 4-3                       | Phase 4-3 migration designed + Codex-cleared + applied to staging first (no production credentials present yet) + key management runbook committed + production apply gated behind Phase 4-20 GO + Phase 4-8 production migration runbook.                             |
| 5   | **Production observability + alerting**  | Phase 3-11's signal catalog was authored for staging; production needs alerting wired to AiSHA's production on-call channel (Slack channel, PagerDuty/Opsgenie if present, escalation routing). Without production observability the pilot operates blind to runtime failures, dead letters, projection lag, adapter latency, and provider-side errors. Phase 4-4 packet adapts the Phase 3-11 signal catalog for production (likely with a lower IMPLEMENTED count because production constraints differ — PARTIAL / DEFERRED items must be explicitly acknowledged before activation, not silently dropped).                                                                                                                 | Phase 4-4                       | Phase 4-4 design freeze documents the production signal catalog + alert routes + on-call rotation (if any) + the explicit list of PARTIAL/DEFERRED signals; the downstream operational observability runbook wires alert routing; Phase 4-20 accepts the residual gap. |
| 6   | **Production rollback plan**             | Phase 3 used the same person (Dre) for deploy + rollback because staging risk is low. ### 10.3 Cross-packet rules

- **The per-packet `depends on` column in §10.2 is the authoritative scheduling rule.** Bands are an approximate summary; if a packet's dependencies are satisfied, that packet may be authored regardless of which band it nominally belongs to.

Codex enforces the distinction on each review.
- **No Phase 4-N packet may reference code that has not been implemented when it implies a runtime state.** A runbook that documents "execute step X against committed code Y" must have Y already implemented + Codex-cleared (this is what distinguishes "design freeze" dependencies from "implementation" dependencies above).
