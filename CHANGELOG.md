### Fixed

- **Contacts customer-name search no longer fails to load records in V2 list filtering (4VD-61) (`backend/routes/contacts.v2.js`, `backend/__tests__/routes/v2-inline-stats.test.js`)**: The contacts page builds a multi-field `$or` search that includes customer/company terms, but the V2 contacts route was forwarding arbitrary `$icontains` fields directly into Supabase. When the filter included `company`, the backend could error instead of returning matching contacts, which surfaced in the UI as "Failed to load contacts" when users typed a customer name. The V2 route now whitelists valid contact search fields, translates `company` / `account_name` lookups into tenant-scoped account ID matches, and safely handles nested `$and` filter groups produced by the contacts page. Regression coverage was added for company-name search, and the test helper was narrowed so the new V2 regression path does not send legacy `created_date` / `updated_date` fields into the V2 create routes.

- **`usePersistedFilters` rehydrates when `storageKey` changes after mount (4VD-59 follow-up, Codex review) (`src/hooks/usePersistedFilters.js`, `src/hooks/__tests__/usePersistedFilters.test.js` (+4 cases, now 17 total))**: The original implementation only read sessionStorage in the `useState` initializer, so when the hook mounted with a `null` key (tenant/user context not yet resolved) and the key later flipped to a real value, the saved filters were never restored. Same bug on tenant switch: previous in-memory filters leaked across keys until manually changed or reset, breaking the intended per-tenant/user isolation. Added a `useEffect` keyed on `storageKey` that compares against a `hydratedKeyRef` and re-reads from sessionStorage (or resets to defaults if the new key isn't ready, e.g. on logout). New tests cover: null→real key transition rehydrates from existing sessionStorage entry, real→real key transition (tenant switch) rehydrates from the new key's saved state, real→null transition (logout) resets to defaults, and null→real transition with no saved state under the new key falls back to defaults.

- **Filter state persistence across navigation for all 7 entity list pages (4VD-59) (`src/hooks/usePersistedFilters.js` (NEW), `src/hooks/__tests__/usePersistedFilters.test.js` (NEW, 13 cases), `src/pages/Leads.jsx`, `src/pages/Contacts.jsx`, `src/pages/Accounts.jsx`, `src/pages/Opportunities.jsx`, `src/pages/Activities.jsx`, `src/pages/BizDevSources.jsx`, `src/pages/Customers.jsx`)**: Navigating to a detail panel and back no longer resets the user's search/filter/sort state. Each page's `handleClearFilters` now delegates to `resetFilters()` rather than resetting individual states. All 13 test cases pass; full suite: 88 files / 821 tests, 0 failures.

Pattern: when `recentlySubmitted=true`, interval switches to `shortPollMs` and a timer begins; after `shortWindowDurationMs` (default 30s), interval reverts to `longPollMs` and the timer clears. Caller re-checks on every tick, so `recentlySubmitted` state changes are picked up immediately. **useSigningSessions refactor**: imported `useAdaptivePoll` and replaces the static `pollMs` parameter with three new optional params (`shortPollMs`, `longPollMs`, `recentlySubmitted`). The hook calculates `currentInterval` via `useAdaptivePoll` and uses it in the fetch polling loop — when interval changes, the effect re-runs and the new interval takes effect on the next tick. **Detail panel integration** (Contact, Lead, Account, Opportunity): added local state `const [recentlySubmittedDoc, setRecentlySubmittedDoc] = useState(false)`, a handler `const handleDocumentSent = () => { setRecentlySubmittedDoc(true); }` that does NOT manually reset the flag (the 30s auto-timeout in useAdaptivePoll handles revert), wired the handler to SendDocumentDialog's `onSent` callback, and passed `recentlySubmitted: recentlySubmittedDoc` to useSigningSessions. Result: first 30s after a document send, panels poll at 5s (6 req/min per panel, catching async pipeline completion within one cycle); steady-state polls at 15s (2 req/min per panel, 6 req/min across 3 open panels). **Test coverage**: `useAdaptivePoll.test.js` pins 7 cases covering default behavior (long-poll in steady-state), switching to short-poll when flag becomes true, auto-revert after window expires, custom intervals, rapid flag toggling, timeout cleanup on flag-false, and complex transitions. `useSigningSessions.test.js` verifies the hook accepts the new props, integrates the adaptive timer correctly, preserves manual refresh independent of cadence, and handles enable/disable transitions. - **eSign engine day 5 PR 1: pdf-lib stamping + auto-date + Certificate of Completion (4VD-43) (`backend/lib/signPdf.js` (NEW), `backend/lib/buildCertificateOfCompletion.js` (NEW), `backend/lib/finalizeSigningSession.js` (NEW), `backend/routes/public-sign.js` (wired finalize), `backend/package.json` (+pdf-lib@1.17.1), `backend/__tests__/lib/signPdf.test.js` (NEW, 16 cases), `backend/__tests__/lib/buildCertificateOfCompletion.test.js` (NEW, 17 cases), `backend/__tests__/lib/finalizeSigningSession.test.js` (NEW, 11 cases))**: Closes the legal/admissibility loop on the in-house eSign engine. Until now, recipients could submit signatures and the audit row was correctly recorded, but no signed PDF artifact existed — making the engine unshippable for actual customer use. **`signPdf.js`** is a pure stamping pipeline: takes the original template PDF + template field definitions + recipient-supplied values + the server-side `signed_at` timestamp, returns a stamped PDF Uint8Array. Field semantics: `signature` embeds the recipient's PNG (preserves aspect ratio, centered in the box), `text`/`name`/`email` draw with Helvetica sized to fit area height (clamped 6-14pt, top-aligned), `checkbox` draws an X centered in the box when truthy, `date` ALWAYS stamps the server `signed_at` formatted `MM/DD/YYYY` and DISCARDS any recipient-typed value (legal admissibility — the date on a contract is the date of execution, not whatever the signer typed; ESIGN Act / eIDAS standard). Coordinate conversion handles top-left-origin normalised areas → pdf-lib's bottom-left-origin pixel space. PDF metadata embeds the signer name in Author and the ISO timestamp in Subject for downstream forensic traceability. Output saved with `useObjectStreams: false` so the PDFs are maximally compatible with all PDF readers + text-extraction tools (pdf-lib's default object streams trip pdf.js forks during decompression). **`buildCertificateOfCompletion.js`** generates the legally admissible CoC page that gets appended to every signed PDF before storage. Single-page US-Letter layout: header band, document section (name + envelope ID + SHA-256 of original), sender section (tenant + sending user), recipient section (typed name + email + IP + UA + signed-at), audit-trail table (action / timestamp / IP / UA, one row per audit entry, capped at 30 with a "+N earlier event(s)" footer for >30), and a closing legal footer. The SHA-256 hash binds the CoC to the specific original PDF — re-hash the embedded original to verify integrity later. `appendCertificateOfCompletion` convenience function: takes a signed PDF buffer + CoC params, returns merged Uint8Array via pdf-lib's `copyPages`. **`finalizeSigningSession.js`** orchestrates the post-submit pipeline: re-loads the session row + parent template, downloads the original PDF from `tenant-assets/{tenant_id}/templates/{template_id}.pdf`, computes SHA-256, stamps fields, builds and appends CoC, uploads merged PDF to `tenant-assets/{tenant_id}/signed/{session_id}.pdf`, updates `signing_sessions` with `signed_pdf_storage_path` + `status='completed'` + `completed_at` + a `completed` audit-trail entry. **Idempotent re-run**: calling finalize on a row already at `status='completed'` returns ok+`already_completed` without a second upload (operator-visible re-runs via console / one-off script are safe). **Best-effort write-behind**: the recipient gets their success page back from `POST /api/sign/:token/submit` IMMEDIATELY (status='signed' is the legal intent-to-sign record); the finalize pipeline runs un-awaited so pdf-lib + storage upload don't extend recipient submit latency. Failures log + return structured `{ok:false, reason}`; the row stays at `status='signed'` and an operator can re-run finalize manually. **Wire-in** (`public-sign.js` POST submit): added `finalizeSigningSession({...}).then(...)` immediately after `updateActivityForSign` — same fire-and-forget pattern, no behavior change to the recipient's response shape. **Test coverage** (44 cases across the three lib modules): pure helpers (date formatting + tz, PNG data-URL decode, font-size fit), round-trip PDF parses (page count, metadata round-trips), the legal invariant that date fields stamp `signed_at` not the recipient-typed value (with both metadata and pdf-parse text-layer assertions), failure modes (garbage PDF, missing args, unknown field types silently skipped, out-of-range page index, supabase errors at every pipeline step), append-page-count + page-order assertions on CoC merge, idempotent re-run, sha256Hex against Node crypto vectors. Tests use `pdf-parse` (already in deps) for text-layer extraction since pdf-lib's FlateEncode-compressed content streams aren't grep-able from raw bytes. **Out of scope for this PR (queued for PR 2)**: Type-signature option in `SignaturePad.jsx`, `GET /api/submissions/:id/signed-pdf-url` admin download endpoint, public recipient download endpoint, recipient auto-email with the signed PDF attached. **In progress towards 4VD-43.**

- **Auto-sync `activities.related_name` + `related_email` on entity rename (4VD-43 polish, applies CRM-wide) (`backend/migrations/167_sync_activities_related_name.sql`, applied to dev `nrtrjsatmsosslxwlmoj` and staging `bjedfowimuwbcnruwcdj`)**: When a lead/contact/account/opportunity is renamed (or its email changed), every existing activity row referencing that entity now updates its denormalised `related_name` + `related_email` automatically. Without this, a "Document sent — Service Agreement" activity originally tagged "Test Lead" would still read "Test Lead" months later after the lead became "Confirmed Customer (CTO)" — stale data in the timeline. **Implementation**: single shared trigger function `public.sync_activities_related_name()` dispatching on `TG_ARGV[0]` (entity_type), with one `AFTER UPDATE` trigger per entity table — `sync_lead_name_to_activities`, `sync_contact_name_to_activities`, `sync_account_name_to_activities`, `sync_opportunity_name_to_activities`. WHEN clauses limit firing to actual rename / email-change events (`first_name`/`last_name`/`email` for leads + contacts; `name`/`email` for accounts; `name` only for opportunities since the table has no email column), so high-frequency churn columns (lifecycle, scoring, lead_score, status, dates) don't cascade into activity updates. Function uses `SECURITY DEFINER` + explicit `search_path = public, pg_temp` so it can update activities even when the triggering session is a tenant-scoped user (RLS would otherwise block the cross-row update); tenant isolation is enforced by the explicit `WHERE tenant_id = NEW.tenant_id` filter inside the function. Index path is the existing `idx_activities_related_id (tenant_id, related_id) WHERE related_id IS NOT NULL`. **Why a Postgres trigger over route-level sync**: catches every update path uniformly (REST routes, Braid tools, AI agents, future migrations, direct SQL); engine-side enforcement is unambiguous; cost is bounded since entity renames are a rare event class. **Smoke test on dev**: synthetic activity inserted referencing a real lead → lead renamed via UPDATE → activity's `related_name` re-read shows the new composed name → revert + cleanup, all within a single PL/pgSQL DO block. **Out of scope (deferred)**: bizdev_sources — its display name lives in `company_name`/`contact_person` not `first_name`+`last_name`, AND `resolveRelatedEntityFields.js` is also miswired for that entity (declares `nameFields: ['first_name','last_name']` against a table that has neither). - **SignPage: render fields in visual reading order so Tab navigation does not skip backward across the page (4VD-43 day 4b root-cause fix) (`src/pages/SignPage.jsx`, `src/pages/__tests__/SignPage.smoke.test.jsx`)**: Reported symptom was "page opens at the signature block instead of the top". After three rounds of progressively more aggressive scroll-on-mount overrides (none of which worked), live DOM inspection via Claude-in-Chrome found the actual cause: fields were rendering in template-declaration order, NOT visual reading order. On the Service Agreement template, `name_3` (right column on page 2) was added to the template before `signature_4` (left column, same row). DOM order was `text_1 → text_2 → name_3 → signature_4 → date_5`, which meant Tab from `text_2` (page 1) jumped past the visually-prior `signature_4` straight to `name_3` (bottom of page 2 right column), dragging the page-scroll with focus. The "lands at signature block" perception was Tab-induced focus-scroll, not a mount-time bug. **Fix**: in `PdfWithFields`, the `fieldsOnPage` array is now `.slice().sort(...)`-ed by (y, x) before render. Comparator: if `Math.abs(aArea.y - bArea.y) > 0.02` (≈2% of page height) sort by y, otherwise sort by x — the row-tolerance keeps two same-baseline fields (e.g. a "BY: **_ / Print name: _**" pair) in left-then-right order rather than oscillating on sub-pixel y noise. Result: DOM order matches visual reading order, Tab walks the form naturally. **Date input swap**: Claude-in-Chrome also flagged that the native `<input type="date">` traps focus inside its month/day/year sub-fields — Tab cycles internally and never escapes to the next form field. Replaced with `<input type="text" inputMode="numeric" placeholder="MM/DD/YYYY" pattern="\d{1,2}[/-]\d{1,2}[/-]\d{2,4}">` so Tab-out works normally; pdf-lib stamping (day 5) will parse whatever string the recipient enters, same as DocuSeal's date-field semantics. **Scroll override simplification**: stripped the previous useLayoutEffect down to the minimal version (capability check + `scrollRestoration='manual'` + one-shot `scrollTo(0,0)` + cleanup). Removed: the 1.5-second hard scroll-lock with userScrolled detection, the requestAnimationFrame + 50ms/250ms/800ms/1400ms retry chain, the wheel/touchstart/keydown/mousedown/scroll listeners, all `[SignPage scroll]` console diagnostics. Those were solving a problem that didn't exist; keeping the minimal scrollRestoration override is just hygiene for refresh UX. **Regression tests**: replaced the four scroll-lock tests with two focused tests on the new behavior — `(y, x)` sort with row tolerance + the date-input substitution (no `type="date"`, presence of `MM/DD/YYYY` placeholder + numeric inputMode). The minimal `scrollRestoration='manual'` + `useLayoutEffect` tests stay. **In progress towards 4VD-43.**

- **SignPage: disable browser scroll restoration so the public sign URL always starts at the top (4VD-43 day 4b polish) (`src/pages/SignPage.jsx`, `src/pages/__tests__/SignPage.smoke.test.jsx`)**: When a recipient revisits or refreshes a `/sign/<token>` URL, modern browsers restore the previous scroll position by default — the recipient lands halfway down the contract, missing the document title and the floating "Next required field" guidance bar above the fold. **Fix moved to `useLayoutEffect`** so the override runs synchronously after DOM mutation but BEFORE the browser paints — earlier than `useEffect`, which only fires post-paint and would let the user briefly see the page at the browser-restored position. Setup: capability-checks `'scrollRestoration' in window.history` and SSR/older-browser-guards (`typeof window === 'undefined'`), saves the previous mode, sets it to `'manual'`, then runs a `forceTop()` synchronously and again at +50ms / +250ms / +800ms via `setTimeout`. The chain bails out the moment the user takes control: `wheel` / `touchstart` (passive) / `keydown` (capture) listeners flip a `userScrolled` sentinel and `forceTop()` early-returns when set — we don't want to fight a recipient who's deliberately read down the doc.
